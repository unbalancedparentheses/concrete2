Concrete

A systems language for code that must stay understandable under audit.

Concrete is built around a simple claim: semantically important program facts should stay visible. Authority should stay visible in function signatures. Ownership and cleanup should stay visible in value movement. Unsafe implementation techniques should stay behind explicit trusted boundaries instead of diffusing through ordinary code. The goal is not maximal convenience. The goal is a language whose behavior, authority surface, and trust boundaries are easier to review mechanically and easier for humans to audit.

What Concrete Optimizes For

  • Auditability: important semantic facts should remain legible instead of disappearing behind abstraction layers
  • Visible authority: capabilities stay part of the program surface rather than collapsing into ambient authority
  • Visible ownership: allocation, transfer, and cleanup stay explicit enough to inspect
  • Explicit trust boundaries: safe code, trusted internals, foreign unsafety, and proof-oriented subsets stay separate and named
  • Small understandable runtime: no GC, no hidden finalization, no large managed-runtime model
  • Mechanically reviewable evidence: compiler reports for capabilities, unsafe, allocation, proof, and high-integrity questions

What Concrete Is Not

  • Not convenience-first: it does not try to erase ownership, authority, or cleanup costs for the sake of surface ergonomics
  • Not a hidden-runtime language: it is not moving toward GC, invisible finalization, or background semantic work outside the code’s visible execution model
  • Not abstraction-maximizing: features have to earn their place by improving clarity or capability, not by making code look shorter in isolation
  • Not “safe by vibes”: the project separates safe code, trusted implementation techniques, and foreign unsafety explicitly
  • Not a proof-only project: proofs matter, but they sit on top of a practical hosted systems language rather than replacing one
